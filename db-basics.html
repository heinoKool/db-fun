1. Die Grundlagen: Was ist was?

Datenbank (DB)

Die Datenbank ist im Grunde nur die reine, strukturierte Sammlung von Daten. Stellen Sie sich ein riesiges, perfekt organisiertes Lagerhaus voller Akten (Daten) vor. Das Lagerhaus selbst *tut* nichts, es *beinhert* nur die Informationen.

Datenbankmanagementsystem (DBMS)

Das DBMS ist die Software, die das Lagerhaus verwaltet. Es ist der "Lagerverwalter", der die Akten findet, neue Akten anlegt, alte ändert oder sicherstellt, dass nur berechtigte Personen Zutritt haben. Beispiele sind MariaDB, MySQL, Oracle oder PostgreSQL.

Das 3-Ebenen-Architekturmodell (ANSI-SPARC)

Dieses wichtige Konzept beschreibt, wie ein DBMS die Daten für verschiedene Nutzer "übersetzt". Es trennt die physische Speicherung von der logischen Ansicht. Klicken Sie auf die Ebenen, um mehr zu erfahren.

1. Externe Ebene (Benutzersichten)

Dies ist die Ebene, die der Endbenutzer sieht. Es ist die Benutzeroberfläche, eine GUI, ein Formular oder ein Bericht. Jeder Benutzer (z.B. ein "Sachbearbeiter" oder ein "Administrator") sieht nur die Daten, die für ihn relevant sind, in einer für ihn aufbereiteten Form. Er muss nicht wissen, wie oder wo die Daten gespeichert sind.

PDF-Begriffe: GUI, Sachbearbeiter, Administrator

2. Konzeptionelle Ebene (Logisches Modell)

Dies ist die "Architekten-Sicht". Hier wird die logische Struktur *aller* Daten in der Datenbank beschrieben: Welche Tabellen gibt es? Wie hängen sie zusammen (Beziehungen)? Welche Regeln gelten? Diese Ebene ist unabhängig von der physischen Speicherung. Das ER-Modell ist ein Werkzeug, um diese Ebene zu entwerfen.

PDF-Begriffe: Semantisches Modell, ER-Modell, Beziehungen

3. Interne Ebene (Physische Speicherung)

Dies ist die "Maschinenraum"-Ebene. Sie beschreibt, *wie* die Daten tatsächlich auf der Festplatte gespeichert werden. Hier geht es um Dateistrukturen, Indizes, Speicherblöcke und Optimierung. Der normale Entwickler oder Benutzer muss diese Ebene nicht kennen. Das DBMS (z.B. MariaDB) kümmert sich darum.

2. Der Design-Prozess: 4 Phasen

Ein Datenbankmodell entsteht nicht auf einmal. Es ist ein strukturierter Prozess in vier Phasen, der von der vagen Idee bis zur fertigen Implementierung führt. Klicken Sie auf die Phasen für Details.

1. Externe Phase (Anforderungsanalyse)

Was muss das System überhaupt können? In dieser Phase wird der Informationsbedarf der Benutzer ermittelt. Man spricht mit den Anwendern, analysiert bestehende Formulare oder Datenbestände ("Realitätsbeobachtungen", "Benutzersichtanalysen"). Das Ziel ist, zu verstehen, *welche* Daten (Input) gebraucht werden, um das gewünschte Ergebnis (Output, z.B. Berichte) zu erzeugen.

2. Konzeptionelle Phase (Semantisches Modell)

Hier wird der "Bauplan" gezeichnet. Das Ergebnis der Analyse wird in ein formales, aber DBMS-unabhängiges Modell übersetzt. Das wichtigste Werkzeug hierfür ist das **ER-Modell (Entity-Relationship-Modell)**. Es beschreibt die "Dinge" (Entitäten, z.B. `Mitarbeiter`, `Abteilung`) und wie sie zusammenhängen (Beziehungen, z.B. `arbeitet in`).

3. Logische Phase (Relationales Modell)

Der abstrakte ER-Bauplan wird nun in ein konkretes "Tabellenmodell" (relationales Modell) umgewandelt. Aus Entitäten werden Tabellen, aus Attributen werden Spalten. Hier werden auch die Schlüssel (Primary/Foreign Keys) definiert und die Regeln zur Auflösung von m:n-Beziehungen angewendet. Das Ergebnis ist immer noch DBMS-unabhängig (quasi "auf dem Papier").

4. Physische Phase (Implementierung)

Jetzt wird es technisch. Das logische Tabellenmodell wird mit der Sprache **SQL** in einem *spezifischen* DBMS (z.B. MariaDB) implementiert. Hier werden mit `CREATE TABLE` die Tabellen angelegt, Datentypen (z.B. `VARCHAR`, `INT`) festgelegt und die physische Speicherung (z.B. `ENGINE=InnoDB`) definiert.

3. Der Bauplan: Von der Idee zur Tabelle

In der konzeptionellen und logischen Phase wird das Herzstück der Datenbank entworfen. Dies geschieht in zwei Hauptschritten, die wir hier genauer betrachten.

1. Das ER-Modell (Konzeptionell)

Das ER-Modell (Entity-Relationship-Modell) ist der grafische Bauplan. Es zeigt die "Dinge" (Entitäten) und wie sie zusammenhängen (Beziehungen). Die **Kardinalität** (z.B. 1:1, 1:n) beschreibt die *Anzahl* der möglichen Beziehungen.

1:1-Beziehung (Eins-zu-Eins)

Ein Datensatz aus Tabelle A ist mit *genau einem* Datensatz aus Tabelle B verbunden (z.B. `Lehrer` leitet `Klasse`).

1:n-Beziehung (Eins-zu-Viele)

Ein Datensatz aus A kann mit *vielen* Datensätzen aus B verbunden sein (z.B. `Klasse` hat viele `Schüler`).

Viele Datensätze aus A können mit *vielen* Datensätzen aus B verbunden sein (z.B. `Schüler` belegt viele `Kurse`).

Wichtig: m:n-Beziehungen können nicht direkt in Tabellen umgesetzt werden. Sie *müssen* aufgelöst werden (siehe Tabellenmodell).

3. Schlüssel & Regeln

Schlüssel sind das Fundament, das die Tabellen zusammenhält. Sie garantieren Eindeutigkeit und Konsistenz.

Visuelles Beispiel: PK & FK

Der Primärschlüssel (PK) identifiziert eine Klasse eindeutig. Der Fremdschlüssel (FK) in der Schülertabelle stellt die Verbindung her, indem er auf den PK der Klasse verweist.

Tabelle: Klasse
id (PK)Name
110a
210b
Tabelle: Schüler (zeigt auf Klasse)
id (PK)Namekid_FK (FK) ←
101Max1
102Anna2
103Tim1
Primärschlüssel (Primary Key, PK)

Ein Attribut (oder eine Kombination), das einen Datensatz (Zeile) in einer Tabelle *eindeutig* identifiziert. (z.B. `Schüler.id`).

  • Darf nicht `NULL` (leer) sein.
  • Muss eindeutig sein.
  • Oft ein "künstlicher" Zählschlüssel (z.B. `AUTO_INCREMENT`).
Fremdschlüssel (Foreign Key, FK)

Ein Attribut in einer Tabelle (z.B. `Schüler.kid_FK`), das auf einen *Primärschlüssel* in einer anderen Tabelle (z.B. `Klasse.id`) verweist. Dies ist die technische Umsetzung einer "Beziehung".

  • Stellt die Referentielle Integrität sicher: Es kann keinen `Schüler` mit einer `kid_FK` geben, die nicht in der `Klasse`-Tabelle als `id` existiert.
Verbundschlüssel (Composite Key)

Ein Primärschlüssel, der aus *mehreren Spalten* gleichzeitig besteht. Die Kombination aller Spalten muss eindeutig sein.

  • Typischer Anwendungsfall: Zwischentabellen bei m:n-Beziehungen.
Beispiel: Tabelle 'belegt'
schueler_id (PK, FK)kurs_id (PK, FK)
10142
10143
10142

Die letzte Zeile wäre ungültig, da die Kombination (101, 42) bereits existiert.

4. Daten-Hygiene: Normalisierung

Warum der ganze Aufwand mit dem Aufteilen in Tabellen? Um Redundanz (doppelte Daten) zu vermeiden. Redundanz führt zu "Anomalien" (Fehlern) bei Änderungen, Löschungen oder beim Einfügen von Daten. Die Lösung ist die **Normalisierung**.

5. Die Sprache: SQL

SQL (Structured Query Language) ist die Sprache, mit der wir mit der Datenbank "sprechen". Sie gliedert sich in mehrere Teilbereiche. Klicken Sie auf die Karten, um die wichtigsten Befehle zu sehen.

DQL

Data Query Language

Der wichtigste Teil: Daten abfragen und lesen.

SELECT * FROM MITGLIEDER
WHERE Name = 'Maier'
ORDER BY Geburtsdatum;

DML

Data Manipulation Language

Befehle, um Daten zu ändern, einzufügen oder zu löschen.

INSERT INTO ort (plz, name) VALUES ('80331', 'München');

UPDATE ort SET plz = '80333' WHERE id = 1;

DELETE FROM ort WHERE id = 1;

DDL

Data Definition Language

Definiert die Struktur der Datenbank selbst. Erstellt oder ändert Tabellen.

CREATE TABLE ort (
    id INT PRIMARY KEY AUTO_INCREMENT,
    plz VARCHAR(10),
    name VARCHAR(50)
);

ALTER TABLE mitarbeiter ADD COLUMN strasse VARCHAR(100);

DROP TABLE ort;

DCL

Data Control Language

Regelt die Zugriffsrechte und Berechtigungen der Benutzer.

GRANT SELECT, UPDATE ON GRUNDBEITRAEGE TO Mueller;

REVOKE DELETE ON GRUNDBEITRAEGE FROM Mueller;

TCL

Transaction Control Language

Steuert Transaktionen – also mehrere Befehle, die als eine Einheit behandelt werden (z.B. eine Banküberweisung).

COMMIT;

ROLLBACK;
